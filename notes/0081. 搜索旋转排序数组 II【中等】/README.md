## 1. 题目描述

- [leetcode](https://leetcode.cn/problems/search-in-rotated-sorted-array-ii/)

已知存在一个按非降序排列的整数数组 `nums`，数组中的值不必互不相同。

在传递给函数之前，`nums` 在预先未知的某个下标 `k`（`0 <= k < nums.length`）上进行了旋转，使数组变为 `[nums[k], nums[k+1], ..., nums[n-1], nums[0], nums[1], ..., nums[k-1]]`（下标从 0 开始计数）。例如， `[0, 1, 2, 4, 4, 4, 5, 6, 6, 7]` 在下标 `5` 处经旋转后可能变为 `[4, 5, 6, 6, 7, 0, 1, 2, 4, 4]`。

给你旋转后的数组 `nums` 和一个整数 `target`，请你编写一个函数来判断给定的目标值是否存在于数组中。如果 `nums` 中存在这个目标值 `target`，则返回 `true`，否则返回 `false`。

你必须尽可能减少整个操作步骤。

---

示例 1：

```txt
输入：nums = [2, 5, 6, 0, 0, 1, 2], target = 0
输出：true
```

---

示例 2：

```txt
输入：nums = [2, 5, 6, 0, 0, 1, 2], target = 3
输出：false
```

---

提示：

- `1 <= nums.length <= 5000`
- `-10^4 <= nums[i] <= 10^4`
- 题目数据保证 `nums` 在预先未知的某个下标上进行了旋转
- `-10^4 <= target <= 10^4`

进阶：

- 此题与 [33. 搜索旋转排序数组][1] 相似，但本题中的 `nums` 可能包含重复元素。这会影响到程序的时间复杂度吗？会有怎样的影响，为什么？

算法思路：

- 与 0033 类似，通过二分查找判断 target 在有序的哪一半
- 区别在于存在重复元素：当 `nums[left] == nums[mid]` 时无法判断有序半边，此时 `left++` 跳过重复值
- 否则根据左半段或右半段有序，判断 target 是否落在有序区间内，收缩边界

## 3. 回答“进阶”部分提到的问题【AI】

重复元素确实会影响算法的时间复杂度，使其从无重复时的 O(log n) 退化为最坏情况下的 O(n)。

### 3.1. 为什么会有影响？

根本原因在于，重复元素的存在破坏了标准二分查找赖以高效运行的“二段性”。

- 无重复时：我们可以通过比较 `nums[left]` 和 `nums[mid]`，明确判断出左半段或右半段是严格有序的，从而安全地排除另一半区间，实现 O(log n) 的对数级效率。
- 有重复时：当 `nums[left] == nums[mid]` 时，我们无法判断哪一半是有序的。比如数组 `[1, 0, 1, 1, 1]` 中，`nums[left]`、`nums[mid]` 和 `nums[right]` 可能都为 1。

在这种情况下，算法唯一的办法就是逐个剔除重复元素，直到打破僵局。这个过程会让算法退化为线性查找。

### 3.2. 最坏情况：全是重复数字

考虑一个最极端的例子：`nums = [1, 1, 1, 1, 1]`，`target = 2`。

1.  算法会检查中间元素，发现是 `1`，不等于 `target`。
2.  同时发现 `nums[left] == nums[mid]`，无法判断哪边有序。
3.  于是执行 `left++`（或 `left++` 和 `right--`），只排除了一个元素。
4.  这个过程会一直持续，直到遍历完整个数组。

由此可见，在元素全部相同的最坏情况下，算法需要访问数组中的每一个元素才能确定 `target` 不存在。

### 3.3. 复杂度总结

- 平均情况：O(log n)。当数组中重复元素较少时，算法仍能保持高效。
- 最坏情况：O(n)。当数组存在大量重复元素时，算法会退化为线性查找。
- 空间复杂度：始终为 O(1)，因为算法只使用了常数级别的额外空间。

### 3.4. 总结

重复元素让算法的最坏情况时间复杂度从 O(log n) 变成了 O(n)。这是因为当遇到无法判断的重复元素时，算法只能通过逐个排除的方式来“清理”障碍，这个过程在最坏情况下需要遍历整个数组。
