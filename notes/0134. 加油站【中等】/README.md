## 1. 题目描述

- [leetcode](https://leetcode.cn/problems/gas-station/)

在一条环路上有 `n` 个加油站，其中第 `i` 个加油站有汽油 `gas[i]` 升。

你有一辆油箱容量无限的的汽车，从第 `i` 个加油站开往第 `i+1` 个加油站需要消耗汽油 `cost[i]` 升。你从其中的一个加油站出发，开始时油箱为空。

给定两个整数数组 `gas` 和 `cost`，如果你可以按顺序绕环路行驶一周，则返回出发时加油站的编号，否则返回 `-1`。如果存在解，则保证它是唯一的。

---

示例 1：

```txt
输入: gas = [1, 2, 3, 4, 5], cost = [3, 4, 5, 1, 2]
输出: 3
```

解释:

- 从 3 号加油站(索引为 3 处)出发，可获得 4 升汽油。此时油箱有 = 0 + 4 = 4 升汽油
- 开往 4 号加油站，此时油箱有 4 - 1 + 5 = 8 升汽油
- 开往 0 号加油站，此时油箱有 8 - 2 + 1 = 7 升汽油
- 开往 1 号加油站，此时油箱有 7 - 3 + 2 = 6 升汽油
- 开往 2 号加油站，此时油箱有 6 - 4 + 3 = 5 升汽油
- 开往 3 号加油站，你需要消耗 5 升汽油，正好足够你返回到 3 号加油站。
- 因此，3 可为起始索引。

---

示例 2：

```txt
输入: gas = [2, 3, 4], cost = [3, 4, 3]
输出: -1
```

解释:

- 你不能从 0 号或 1 号加油站出发，因为没有足够的汽油可以让你行驶到下一个加油站。
- 我们从 2 号加油站出发，可以获得 4 升汽油。 此时油箱有 = 0 + 4 = 4 升汽油
- 开往 0 号加油站，此时油箱有 4 - 3 + 2 = 3 升汽油
- 开往 1 号加油站，此时油箱有 3 - 3 + 3 = 3 升汽油
- 你无法返回 2 号加油站，因为返程需要消耗 4 升汽油，但是你的油箱只有 3 升汽油。
- 因此，无论怎样，你都不可能绕环路行驶一周。

---

提示：

- `n == gas.length == cost.length`
- `1 <= n <= 10^5`
- `0 <= gas[i], cost[i] <= 10^4`
- 输入保证答案唯一。

- 时间复杂度：$O(n)$，其中 $n$ 是加油站的数量
- 空间复杂度：$O(1)$，只使用了常数级别的额外空间

算法思路：

- 维护 `totalTank` 记录总油量差，若总油量差 < 0 则无解
- 维护 `curTank` 记录从当前起点 `start` 出发的累计油量
- 若 `curTank < 0`，说明从 `start` 出发无法到达 `i`，将起点更新为 `i + 1` 并重置 `curTank`

### 2.1. 问题：为什么 `totalTank >= 0` 就一定有解？【AI】

这个问题问到了这道题最核心的数学原理。简单来说：**`totalTank >= 0` 是“能绕一圈”的充要条件（必要且充分）。**

为了让你彻底信服，我从**数学证明**和**算法逻辑**两个层面来解释：

#### 数学必然性（全局视角）

把每个加油站的“净收益”记为 `diff[i] = gas[i] - cost[i]`。

- **必要性**：绕一圈总共加的油必须 >= 总共消耗的油，否则油箱必然见底，所以 `totalTank >= 0` 是必须的（否则返回 -1）。
- **充分性（为什么够了）**：如果所有 `diff` 的总和 `>= 0`，那么**必然存在**一个起点，使得从该点出发的任意前缀和都不小于 0。这是一个经典的数学结论（类似于“环状数组最大子段和”），可以反证：如果所有起点的前缀和都会在某处小于 0，那么整个数组的总和不可能大于等于 0。

::: tip 💡 补充说明

上述推断的结果：

- 结果1. 能绕一圈 => `totalTank >= 0`
- 结果2. `totalTank >= 0` => 能绕一圈（且至少一个解）

在题目描述中有这么一句话：“如果存在解，则保证它是唯一的”，加上结果 2，就能推断出：**`totalTank >= 0` 意味着一定有解，且解是唯一的**。

:::

---

#### 代码中的贪心逻辑（“断点重置”的原理）

代码中的 `curTank` 和 `start` 重置机制，恰好帮我们**找到了那个必然存在的起点**。

关键逻辑在这里：

```javascript
if (curTank < 0) {
  start = i + 1 // 舍弃 [start, i] 这一段
  curTank = 0
}
```

当 `curTank` 变成负数时，意味着**从当前的 `start` 到 `i` 这段区间内，净油量总和是负数**。

**为什么起点可以放心地跳到 `i+1`？**
因为 `start` 到 `i` 之间的**任意一个点**都不能作为起点。反证一下：

- 假设从 `start` 出发，到达 `i` 时油量为负（不够）。
- 如果你从 `start` 和 `i` 之间的某个点 `k` 出发，那么到达 `i` 时的油量，**一定比从 `start` 出发到达 `i` 时的油量还要少**（因为你少了 `start` 到 `k` 这段积累的油量）。
- 既然从 `start` 出发都撑不到 `i+1`，从中间出发更撑不到。所以 **`start` 到 `i` 全都废了**，起点只能在 `i+1` 之后。

---

#### 结合 `totalTank >= 0` 证明最终的 `start` 有效

我们遍历数组时，把所有“拖后腿”的负段（`curTank < 0` 的区间）都切掉并丢弃了。
假设我们一共切掉了 `k` 段，它们的总净收益分别是 `S1, S2, ..., Sk`，并且**每一段都是负数**（因为触发重置时的 `curTank` 就是该段的和）。

最终剩下的那段（从最后的 `start` 到数组末尾）的净收益记为 `S_last`。

因为：
`S1 + S2 + ... + Sk + S_last = totalTank >= 0`
且 `S1, S2, ..., Sk` 都是负数，
所以 **`S_last` 必然大于等于 0，并且足够大，能抵消前面所有负段的总和**。

当汽车从最终的 `start` 出发，先走完 `S_last` 这一段（油箱一直非负），然后绕回开头，依次经历之前被切掉的负段 `S1, S2...`。因为 `S_last` 存下了足够多的油，正好用来弥补那些负段的亏空，所以最终能顺利回到起点。

---

**总结一句话：**
`totalTank >= 0` 保证了“好油”总量能覆盖“坏路”总消耗，而贪心切负段的操作，正好把最难的“坏路”留到了最后，用前面攒下的油去抵消，所以最终 `start` 必然有效。这也是为什么最后直接 `return start` 即可，无需再次验证。
